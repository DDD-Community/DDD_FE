# 다음 할 일 브리핑

당신은 DDD 프론트엔드 레포에서 "다음에 무엇을 해야 하는가"를 간결하게 브리핑한다. 기준 문서는 **`tasks/checklist.md`** 이며, 현재 브랜치 상태도 함께 고려한다.

## 입력

- 인자: `$ARGUMENTS`
  - 비어 있으면 전체 기획서에서 우선순위 도출
  - 키워드(`semesters`, `reminders`, `projects`, `blog`, `applications`, `/3.1`, `인증` 등)가 있으면 해당 섹션만 좁혀서 브리핑

## 절차 (고정)

1. **기준 문서 로드**
   - `tasks/checklist.md` 를 읽어 ✅ / 🔧 / ⬜ 항목을 파악한다.
   - 필요 시 `CLAUDE.md`, `CODE_RULES.md` 에서 도메인·규약을 확인한다.
2. **현재 상태 스캔**
   - `git status`, `git log --oneline -5`, `git diff --stat`(변경이 있을 경우)로 in-flight 작업을 파악한다.
   - 인자가 주어지면 해당 도메인 폴더(`apps/admin/src/pages/<domain>/` 등)를 Glob/Grep 으로 확인해 실제 구현 상태가 체크리스트와 일치하는지 빠르게 대조.
3. **우선순위 도출 규칙**
   - P0: 인증/라우트 보호 등 **블로커** 성격 (⬜ 중에서도 다른 작업을 막는 것)
   - P1: 현재 브랜치에서 진행 중(🔧)인 항목의 남은 일
   - P2: 기획서에서 다음 단계로 자연스러운 ⬜ 항목 (의존성·도메인 연관성 고려)
   - P3: Nice-to-have / 정리·리팩터
   - `/tasks/checklist.md` 의 "공통 인프라" 이슈는 도메인 작업보다 우선한다.
4. **출력** — 아래 고정 포맷. 군더더기 없이 간결하게.

## 출력 포맷

```
## 🧭 다음 할 일 브리핑
**기준**: tasks/checklist.md ({{섹션 또는 '전체'}})
**브랜치**: {{current branch}} · 변경 {{N}}개 파일

### 🔥 지금 이어서 (In-flight)
- [checklist §X.Y] 항목 — 남은 일: …
  - 관련 파일: path/to/file.tsx

### ⏭️ 다음 추천 (P1 → P3)
1. **P1** [§X.Y] 항목 — 이유/의존성 한 줄
2. **P2** [§X.Y] 항목 — …
3. **P3** [§X.Y] 항목 — …

### 🧱 블로커 / 선결 조건
- 예: 인증 보호 라우트가 없어 `/applications` 후속 작업이 막힘

### ✅ 바로 착수할 1개 (추천)
- **무엇**: …
- **왜**: …
- **첫 수**: 파일/명령/컴포넌트 시작점 제시
- **완료 정의**: 체크리스트 상 어떤 항목이 ✅ 로 바뀌어야 하는가
```

## 행동 규칙

- **직접 코드를 수정하지 않는다.** 이 커맨드는 브리핑 전용.
- 체크리스트에 없는데 변경이 일어난 부분이 있으면 "범위 외"로 표기하고 알린다.
- 한국어로 응답한다.
- 전체 응답은 **터미널 한 화면**에 들어가는 분량을 목표로 한다(불필요한 배경 설명 생략).
- 추측하지 말고, 근거가 되는 **파일:라인 / 체크리스트 §번호**를 함께 제시한다.
